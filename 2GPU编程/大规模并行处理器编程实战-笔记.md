### 2.5核函数和线程

核函数：在并行阶段，所有线程将要执行的函数。每个线程执行的是相同函数，因此CUDA C是SPMD（单一程序多数据）编程风格的实例

当主机端调用一个核函数，所有的线程被组织为两级结构：

一个核函数由一个 grid 运行，一个 grid 含有多个 blocks，一个 block 内有多个 threads。每个 block 内 threads 的数量都是相同的，且最多为 1024 个。

blockIdx，blockDim，threadIdx

x，y，z三个维度，根据数据维度来组织线程维度，每个维度线程数量应为32的倍数

void vecAdd(float* A, float* B, float* C, int n) {
    float *A_d, *B_d, *C_d;
    int size = n * sizeof(float);

    cudaMalloc((void **) &A_d, size);
    cudaMalloc((void **) &B_d, size);
    cudaMalloc((void **) &C_d, size);

    cudaMemcpy(A_d, A, size, cudaMemcpyHostToDevice);
    cudaMemcpy(B_d, B, size, cudaMemcpyHostToDevice);

    vecAddKernel<<<ceil(n/256.0), 256>>>(A_d, B_d, C_d, n);

    cudaMemcpy(C, C_d, size, cudaMemcpyDeviceToHost);

    cudaFree(A_d);
    cudaFree(B_d);
    cudaFree(C_d);
}

```c
__global__ void blurKernel(unsigned char *in, unsigned char *out, int w, int h) {
    // 每侧的像素数量, 2 * BLUR_SIZE + 1 = 某维度的总像素数量
    int BLUR_SIZE = 1;
    int col = blockDim.x * blockIdx.x + threadIdx.x;
    int row = blockDim.y + blockIdx.y + threadIdx.y;
    if (col < w && row < h) {
        int pixVal = 0; // sum
        int pixels = 0; // cnt
        for (int blurRow = -BLUR_SIZE; blurRow < BLUR_SIZE + 1; ++blurRow) {
            for (int blurCol = -BLUR_SIZE; blurCol < BLUR_SIZE + 1; ++blurCol) {
                int curRow = row + blurRow;
                int curCol = col + blurCol;
                if (curRow >= 0 && curRow < h && curCol >= 0 && curCol < w) {
                    pixVal += in[curRow * w + curCol];
                    ++pixels;
                }
            }
        }
        out[row * w + col] = (unsigned char)(pixVal / pixels);
    }
}
```

## Chapter4,计算架构和调度

### 4.1现代GPU架构

GPU并不是一个巨大的处理器，而是由多个**流式多处理器（Streaming Multiprocessor, SM）**组成的阵列。每个SM是相对独立的执行单元，可以同时运行多个线程块。

流式多处理器(SM)包含多个处理单元(核心)

![image-20260424162109107](D:\proj\LittelWidgets\2GPU编程\大规模并行处理器编程实战-笔记.assets\image-20260424162109107.png)

#### SM内部结构

每个SM包含：

1. **CUDA核心**：执行整数和单精度浮点运算
2. **Tensor核心**：专用矩阵运算（深度学习加速）
3. **特殊功能单元（SFU）**：执行sin/cos/exp等超越函数
4. **加载/存储单元（LD/ST）**：处理内存访问
5. **Warp调度器**：调度线程执行
6. **寄存器文件**：快速存储，每SM约64KB
7. **共享内存**：block内共享的可编程缓存

关键认识：SM是资源池，多个block共享这些资源。block分配多少资源，决定了SM能同时运行几个block。

### 4.2调度

### 4.3同步和透明可扩展性

屏障同步:保证每个线程都执行完毕,执行快的需要等待慢的

一个线程块中的所有线程需要被分配到同一个流式多处理器SM上,一个线程块只有在运行是系统确保了所有线程所需的全部资源准备好才会开始执行

CUDA在设计时候的折中,不允许跨block的屏障同步

透明可扩展性:同一份代码,在不同设备里面执行,速度可以不一样,需要的资源也可以不一样

### 4.4线程束和SIMD硬件

SIMD(single-instruction-multiple-data):单指令多数据模型

warp:线程束, 在目前大部分的设备中，一个 block 内的线程会按照 32 为一个单位组成线程束，SM 内部的调度以线程束为单位进行。如果剩余线程不满 32 个，则会填补一些非激活线程凑满一个线程束。

线程束是线程调度的基本单位,执行的基本单位

SIMT单指令多线程,类似与SIMD,但更灵活,线程可以有独立状态

### 4.5控制流发散*

当warp内的线程走不同分支时，就发生**控制流发散（Control Divergence）**

### 4.6线程束调度和延迟

每个 SM 有多个线程束调度器（如 Ampere 架构有4个）。每个时钟周期，调度器选择一个就绪的线程束发射指令

0开销调度

CPU用复杂的乱序执行、预取来隐藏延迟。GPU用更简单粗暴的方法：**大量线程+快速切换**。

5.一位 CUDA 程序员说，如果他们启动的 kernel 每个块只有 32 个线程，就可以省略需要屏障同步的地方的 `__syncthreads()` 指令。你认为这是个好主意吗？请解释。

**理论上可行，但实践中不推荐。**

**原因分析：**

1. **理论基础**：同一 warp 内的 32 个线程以 SIMT 方式执行，指令级天然同步
2. **实际风险**：
   - 涉及内存操作时，底层硬件可能有延迟，不保证顺序
   - 编译器优化可能重排指令
   - 不同架构行为可能不同
3. **兼容性问题**：
   - NVIDIA 不保证 warp 大小永远是 32
   - 未来架构可能改变执行模型
4. **最佳实践**：
   - 始终使用 `__syncthreads()` 确保正确性
   - 代码更易维护和移植

b. 每块 32 线程，每线程 29 个寄存器

块大小限制: 2048/32 = 64 不满足

寄存器限制:256 * 34 = 8704/块, 65536/8704 = 7块 < 8,不满足

9. 一位学生说他们能够使用 32×32 线程块的矩阵乘法 kernel 来乘两个 1024×1024 的矩阵。该学生使用的 CUDA 设备每块最多支持 512 个线程，每 SM 最多 8 个块。学生还说每个线程计算结果矩阵的一个元素。你的反应是什么？为什么？

**解答：**

**这是不可能的配置，学生的描述存在矛盾。**

**分析：**

1. **结果矩阵元素数**：`1024 × 1024 = 1,048,576` 个元素
2. **学生声称的配置**：
   - 32×32 = 1024 线程/块
   - 但设备限制每块最多 512 个线程！
3. **矛盾点**：32×32 = 1024 > 512，这个块配置本身就超出了设备限制
4. **正确配置示例**：
   - 如果使用 16×16 = 256 线程/块
   - 需要 `1024/16 × 1024/16 = 64 × 64 = 4096` 个块
   - 总线程：`4096 × 256 = 1,048,576`，刚好覆盖所有元素

**注**：每 SM 块数（8 个）对单次 kernel 调用的可行性无影响，只影响并行度和占用率。

### 5.3通过分块Tiling减少访存

共享内存是 Block 内共享的，需要同步保证数据一致性：

使用__syncthreads()函数,不能在某个条件内使用,需要保证Blocks中所有活跃线程都执行到

**核心思想**

既然全局内存慢但共享内存快，策略就是：

1. **分块加载**：将数据分成小块（Tile），逐块加载到共享内存
2. **计算复用**：在共享内存中完成该块的所有计算
3. **移动窗口**：处理下一块，直到完成

这样，每个数据从全局内存只读一次，但在共享内存中被多次使用。

分块可以：

- 减少访问次数
- 利用 shared memory
- 提高性能（通常是数量级提升）

### 5.4分块矩阵核函数

```c
__global__ void matMul(float *M, float *N, float *P, int width) {
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    
    if (row < width && col < width) {
        float sum = 0.0f;
        for (int k = 0; k < width; k++) {
            sum += M[row * width + k] * N[k * width + col];
        }
        P[row * width + col] = sum;
    }
}

```

```c++
// 每个tile(也就是每个block)是16 * 16个线程
// 每个线程计算一个输出元素
#define TILE_WIDTH 16
__global__ void matMulTiled(float *M, float *N, float *P, int width) {
    // 共享内存声明,block内共享
    __shared__ float Mds[TILE_WIDTH][TILE_WIDTH];
    __shared__ float Nds[TILE_WIDTH][TILE_WIDTH];
    
    int bx = blockIdx.x, by = blockIdx.y;
    int tx = threadIdx.x, ty = threadIdx.y;
    
    // 计算该线程负责的 P 元素位置
    int row = by * TILE_WIDTH + ty;
    int col = bx * TILE_WIDTH + tx;
    
    // 分块循环
    float Pvalue = 0;
    for (int ph = 0; ph < width / TILE_WIDTH; ++ph) {
        // 最初的公式还是: row * width + col, 协作加载 M, N 的 Tile
        // 当前行 + 当前 tile 的列
        Mds[ty][tx] = M[row * width + (ph * TILE_WIDTH + tx)];
        // 当前 tile 的行 + 当前列
        Nds[ty][tx] = N[(ph * TILE_WIDTH + ty) * width + col];
        __syncthreads();  // 确保 Tile 加载完成
        // 使用共享内存计算部分点积
        for (int k = 0; k < TILE_WIDTH; ++k) {
            Pvalue += Mds[ty][k] * Nds[k][tx];
        }
        __syncthreads();  // 确保计算完成再加载下一个 Tile
        
    }
    P[row * width + col] = Pvalue;
}

```
因为需要做矩阵乘法,`Pvalue += Mds[ty][k] * Nds[k][tx];`所以M按行加载,N按列加载


| 矩阵 | 加载方式 | 原因               |
| ---- | -------- | ------------------ |
| M    | 按行取   | 因为公式里固定 row |
| N    | 按列取   | 因为公式里固定 col |

```c
Mds[ty][tx] = M[row * width + (ph * TILE_WIDTH + tx)];
Nds[ty][tx] = N[(ph * TILE_WIDTH + ty) * width + col];
```

GPU 关心的是：一组线程（warp）访问的地址是不是连续

Mds按行取,Nds按列取,从线程束warp来看,线程是连续的
